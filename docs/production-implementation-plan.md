# 本実装に向けた要件整理（YouTube Music Playlist Organizer）

このドキュメントは、現状のプロトタイプを本番運用可能なシステムにするために、最初に決めるべき事項をまとめたものです。

## 1. ゴール定義（何を「完成」とするか）

- Googleアカウントでログインし、自分のプレイリストを取得できる
- 左サイドバーでプレイリストを選択できる
- 中央の検索UIから楽曲を検索し、選択中プレイリストに追加できる
- プレイリスト楽曲をドラッグ&ドロップで並び替えできる
- 楽曲削除ができる
- 変更内容をYouTube側へ反映できる（追加・削除・並び替え）
- 失敗時にリトライ可能なエラーハンドリングがある

## 2. API・認証方針（最優先）

### 2.1 認証

- Google Identity Services (OAuth 2.0) を使用
- 必要スコープを最小権限で設計
  - 例: 読み取り専用スコープと編集スコープを分離し、段階的同意を検討
- アクセストークンの扱い方針を確定
  - フロント単体で保持するか
  - バックエンド経由で安全に保管・更新するか

### 2.2 API利用

- YouTube Data API v3 の利用対象を明確化
  - プレイリスト一覧取得
  - プレイリストアイテム取得
  - 追加
  - 削除
  - 並び替え（position更新）
- クォータ消費を最適化する取得・更新戦略を定義
- レート制限/エラーコード別リトライ戦略を定義

## 3. システム構成

### 3.1 推奨構成

- **フロントエンド**: SPA（React / Vue / Svelte いずれか）
- **バックエンド**: BFF（Node.js/TypeScript など）
  - OAuthトークン管理
  - API呼び出しの代理
  - 監査ログ出力
- **データストア（任意）**
  - ユーザー設定（表示設定、検索履歴、並び替え下書き）

### 3.2 なぜBFFを置くか

- トークンをブラウザに長期保持しないため
- APIキー/シークレット露出を防ぐため
- 監視・監査・レート制御を集約するため

## 4. 画面/UX要件

- 初期表示: 左側にプレイリスト一覧、中央に検索UI
- プレイリスト選択前:
  - 左にプレイリスト一覧のみ表示
  - 中央の「追加」操作は無効化またはガイド表示
- プレイリスト選択後:
  - 楽曲一覧表示
  - DnDで順序変更
  - 追加/削除実行
- 保存方式:
  - 即時反映（操作ごとAPI更新）
  - または一括保存（変更を下書き化して最後に保存）
- 状態表示:
  - 進行中（ローディング）
  - 成功（トースト）
  - 失敗（理由 + 再試行）

## 5. データモデル設計

- `User`
  - `id`, `googleAccountId`, `grantedScopes`
- `Playlist`
  - `id`, `title`, `itemCount`, `thumbnailUrl`
- `Track`
  - `videoId`, `title`, `artist`, `duration`, `position`
- `LocalChangeSet`（下書き保存する場合）
  - `added[]`, `removed[]`, `moved[]`

## 6. 並び替え実装の重要ポイント

- UIのドラッグ結果とAPIの `position` 整合を必ず取る
- 複数操作時は競合対策を行う
  - 操作キュー化
  - 連続移動のバッチ更新
- API更新失敗時のロールバック方針を定義

## 7. 検索の実装方針

- 検索ソースを明確化
  - YouTube検索 API を使うか
  - 自前インデックスを用意するか
- 検索条件
  - 曲名 / アーティスト / 部分一致
- UX
  - デバウンス
  - ページング
  - 既存追加済み曲の重複表示制御

## 8. セキュリティ・運用

- OAuthクライアント設定（リダイレクトURI、許可ドメイン）
- CSRF/XSS対策
- 監査ログ（誰がどのプレイリストを更新したか）
- Secrets管理（環境変数 + Secret Manager）
- エラー監視（Sentry等）
- 可用性監視（死活監視、API失敗率）

## 9. テスト戦略

- 単体テスト
  - 並び替えロジック
  - 差分計算
- 結合テスト
  - OAuthフロー
  - プレイリスト取得/更新
- E2Eテスト
  - ログイン→選択→追加→削除→並び替え→保存
- 負荷/耐障害試験
  - レート制限時の挙動
  - APIタイムアウト時の復帰

## 10. 実装ロードマップ（推奨）

1. **要件確定**: スコープ、保存方式、検索方式を決める
2. **認証実装**: Googleログイン + トークン更新
3. **読み取り系実装**: プレイリスト一覧/詳細取得
4. **更新系実装**: 追加・削除・並び替え + エラーハンドリング
5. **UX改善**: ローディング、通知、再試行
6. **品質強化**: テスト自動化、監視導入
7. **段階リリース**: 限定ユーザーで検証後に本番公開

## 11. まず最初の1週間でやること

- Google Cloudプロジェクト準備（OAuth同意画面、認証情報）
- 必要スコープと画面要件の合意
- APIクォータ試算
- BFF雛形作成（認証/トークン管理）
- 現行プロトタイプをコンポーネント分割して移行開始
